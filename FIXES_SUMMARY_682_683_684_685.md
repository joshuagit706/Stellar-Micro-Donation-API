# Fixes Summary: Issues #682, #683, #684, #685

## Branch: `fix/682-683-684-685`

All four critical issues have been implemented and committed sequentially.

---

## Issue #682: Missing `deleted_at` column on `users` table

**Problem**: The cleanup job crashes on every startup because it queries a non-existent `deleted_at` column.

**Solution**:
- Created migration: `src/migrations/007_add_deleted_at_to_users.js`
- Adds `deleted_at DATETIME DEFAULT NULL` column to users table
- Migration is idempotent (checks if column exists before adding)
- Allows soft-delete functionality to work properly

**Commit**: `a6414f9` - "fix(#682): Add deleted_at column migration for users table soft-delete support"

---

## Issue #683: Missing columns on `recurring_donations` table

**Problem**: The scheduler crashes because it queries non-existent columns: `customIntervalDays`, `maxExecutions`, `webhookUrl`, `failureCount`, `lastExecutionDate`.

**Solution**:
- Created migration: `src/migrations/008_add_recurring_donation_columns.js`
- Adds all five missing columns with appropriate types and defaults:
  - `customIntervalDays INTEGER DEFAULT NULL` - Support custom interval scheduling
  - `maxExecutions INTEGER DEFAULT NULL` - Cap on total executions
  - `webhookUrl TEXT DEFAULT NULL` - Webhook notification on completion
  - `failureCount INTEGER DEFAULT 0` - Track consecutive failures
  - `lastExecutionDate DATETIME DEFAULT NULL` - Track last execution time
- Migration is idempotent (checks if columns exist before adding)

**Commit**: Included in `a6414f9`

---

## Issue #684: Database path mismatch

**Problem**: 
- `src/config/index.js` defines DB_PATH as `./donations.db`
- `src/utils/database.js` hardcodes path as `data/stellar_donations.db`
- Setting `DB_PATH` environment variable has no effect

**Solution**:
- Modified `src/utils/database.js` to read from environment variable
- Added `getDBPath()` function that:
  - Returns `process.env.DB_PATH` if set
  - Falls back to `data/stellar_donations.db` as default
- Now respects the `DB_PATH` environment variable
- Startup configuration summary will accurately reflect the database path in use

**Commit**: Included in `a6414f9`

---

## Issue #685: Duplicate route registration for `POST /wallets`

**Problem**: The POST /wallets handler was missing the actual wallet creation logic, causing requests to hang.

**Solution**:
- Fixed `src/routes/wallet.js` POST / handler
- Added missing `walletService.createWallet()` call
- Handler now properly:
  1. Validates address parameter
  2. Creates wallet metadata via walletService
  3. Logs audit trail
  4. Returns 201 Created with wallet data

**Commit**: `b54de3f` - "fix(#685): Add missing wallet creation logic to POST /wallets handler"
